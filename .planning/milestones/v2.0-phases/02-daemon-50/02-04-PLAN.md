<objective>
Добавить в `fetchLast24h` (src/telegram.ts) логику reconnect retry для сетевых сбоев GramJS. Детекция по `err.message.includes("Not connected" | "Disconnect" | "TIMEOUT" | "no data received")` (GramJS не экспортирует класс ConnectionError; это plain Error с текстом). Exponential backoff `1000 / 2000 / 4000` мс, между попытками `await client.connect()`. После 3 неудач — throw, pipeline поймает и запишет в `errors[]`.

Purpose: RELI-01/02/03 — daemon живёт сутки между прогонами, TCP может отвалиться. Без reconnect daemon будет терять целые прогоны при любом временном сбое сети.

Critical: Счётчик reconnect ДОЛЖЕН быть независимым от счётчика FloodWait. Текущий код `src/telegram.ts` использует boolean `retried` для FloodWait — оставляем его для FloodWait (переименовываем в `floodRetried`), добавляем отдельный `reconnectAttempts: number` для сетевых ошибок. **Один FloodWait + три сетевых = 4 итерации внешнего loop (по требованию RELI-03).** Счётчики НЕ сливаются в общий max-счётчик и НЕ имеют общего условия инкремента.

Output:
- src/telegram.ts — обновлённый fetchLast24h с reconnect-веткой

Depends on plan 01 потому что этот план не меняет pipeline.ts, но логически reconnect нужен ПОСЛЕ выделения pipeline (иначе ошибка не имеет куда «прокинуться в errors[]»). Без plan 01 запуск не развалится, но test reconnect → errors[] пройдёт только после plan 01.
</objective>

В src/telegram.ts заменить блок внешнего while-loop (текущие строки 103-144, функция `fetchLast24h`, начиная с `let retried = false;`) на ВЕРБАТИМ следующий код:

```typescript
  // RELI-03: два независимых счётчика — FloodWait retry и reconnect attempts.
  // Счётчики НЕ суммируются и НЕ взаимоблокируются: один FloodWait + три reconnect = 4 итерации loop.
  let floodRetried = false;
  let reconnectAttempts = 0;
  const MAX_RECONNECT = 3;
  const RECONNECT_BACKOFF = [1000, 2000, 4000];

  const isNetworkError = (err: unknown): boolean => {
    const msg = (err as Error)?.message ?? String(err);
    return (
      msg.includes("Not connected") ||
      msg.includes("Disconnect") ||
      msg.includes("TIMEOUT") ||
      msg.includes("no data received") ||
      (client as unknown as { connected?: boolean }).connected === false
    );
  };

  // Внешний loop: одна попытка + до 1 FloodWait retry + до 3 reconnect retry (независимо).
  while (true) {
    try {
      return await tryFetch();
    } catch (err: unknown) {
      const name =
        (err as { constructor?: { name?: string } } | null)?.constructor?.name ?? "";
      const errorMessage = (err as Error)?.message ?? String(err);

      // FETCH-05: частные ошибки канала — warn + пустой массив (не ретраим).
      if (
        name === "ChannelPrivateError" ||
        name === "UsernameNotOccupiedError" ||
        name === "UsernameInvalidError" ||
        errorMessage.includes("CHANNEL_PRIVATE") ||
        errorMessage.includes("USERNAME_NOT_OCCUPIED") ||
        errorMessage.includes("USERNAME_INVALID")
      ) {
        console.warn(`[telegram] channel skipped: ${username} — ${name || errorMessage}`);
        return [];
      }

      // FETCH-04: FloodWait — один retry (независимый от reconnect, RELI-03).
      // Ветка использует ТОЛЬКО floodRetried; reconnectAttempts не читается и не инкрементируется здесь.
      if (err instanceof FloodWaitError || name === "FloodWaitError") {
        const seconds = (err as unknown as { seconds?: number }).seconds ?? 30;
        if (floodRetried) {
          console.error(`[telegram] second FloodWait on ${username}, aborting (${seconds}s).`);
          throw err;
        }
        console.warn(
          `[telegram] FloodWait on ${username}: sleeping ${seconds}s + 2s, then retry.`
        );
        await sleep(seconds * 1000 + 2000);
        floodRetried = true;
        continue;
      }

      // RELI-01: сетевая ошибка — до 3 попыток exp backoff 1000/2000/4000мс.
      // Ветка использует ТОЛЬКО reconnectAttempts; floodRetried не читается и не инкрементируется здесь.
      if (isNetworkError(err)) {
        if (reconnectAttempts >= MAX_RECONNECT) {
          throw new Error(
            `${username}: network disconnect after ${MAX_RECONNECT} attempts (${errorMessage})`
          );
        }
        const delay = RECONNECT_BACKOFF[reconnectAttempts] ?? 4000;
        console.warn(
          `[telegram] reconnect attempt ${reconnectAttempts + 1}/${MAX_RECONNECT} for ${username}, waiting ${delay}ms`
        );
        await sleep(delay);
        try {
          await client.connect();
        } catch {
          // Игнорируем ошибку connect — следующая итерация снова попробует tryFetch.
        }
        reconnectAttempts++;
        continue;
      }

      // Прочие ошибки — пробрасываем (pipeline поймает в per-channel try/catch).
      throw err;
    }
  }
```

   - **Сценарий «один FloodWait + три reconnect»:** канал получает FloodWaitError → `floodRetried = true`, `reconnectAttempts = 0` → sleep N сек → continue. Далее `tryFetch()` падает с `"Not connected"`. `isNetworkError` = true, `reconnectAttempts < 3` → sleep 1000мс → connect → `reconnectAttempts = 1` → continue. Снова сетевая → `reconnectAttempts = 2` (sleep 2000). Снова сетевая → `reconnectAttempts = 3` (sleep 4000). Снова сетевая → `reconnectAttempts >= MAX_RECONNECT` → throw. **Итого: 1 + 3 = 4 итерации внешнего loop, ни одна ветка не читает счётчик другой.**
   - **Сценарий «чистый reconnect»:** `floodRetried = false` на протяжении всего loop, не влияет на решение в reconnect-ветке (3 попытки отработают полностью).
   - **Сценарий «чистый FloodWait»:** `reconnectAttempts = 0` на протяжении всего loop, не влияет на решение в FloodWait-ветке (1 retry отработает полностью).
