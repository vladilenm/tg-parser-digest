   APP_DIR="${APP_DIR:-/opt/tg-parser-demo}"

   echo "[deploy] git pull --ff-only origin main"

- `--ff-only` защищает от случайного merge-коммита если кто-то правил локально на VDS

- НЕ добавлять `npm install` или `npm ci` — это происходит внутри Dockerfile при `--build`.

Создать **новый файл** `.github/workflows/deploy.yml`. Auto-deploy: push в main → ssh на VDS → запуск deploy.sh.

Требования:

```yaml
name: Deploy to Timeweb VDS

on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    # Тихий скип, пока секреты не заданы (например, на этапе подготовки VDS).
    # Когда оператор пропишет VDS_HOST в Settings → Secrets → Actions, workflow
    # начнёт срабатывать автоматически без правки YAML.
    if: ${{ vars.VDS_DEPLOY_ENABLED == 'true' || github.event_name == 'workflow_dispatch' }}
    steps:
      - name: Checkout (для контекста ref в логе)
        uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.2.5
        with:
          host: ${{ secrets.VDS_HOST }}
          username: ${{ secrets.VDS_USER }}
          key: ${{ secrets.VDS_SSH_KEY }}
          port: ${{ secrets.VDS_PORT || 22 }}
          script: bash /opt/tg-parser-demo/deploy.sh
```

ВАЖНО про условный запуск:
- GitHub Actions НЕ позволяет напрямую обращаться к `secrets.X` в `if:` (политика безопасности — нельзя случайно засветить наличие секрета в expression). Поэтому используется `vars.VDS_DEPLOY_ENABLED == 'true'` (repository variable, не секрет — оператор переключит когда настроит VDS).
- Альтернатива: оператор может удалить `if:` совсем после первой настройки секретов — workflow тогда будет работать всегда. Но дефолт «тихий скип через vars» безопаснее.
- `workflow_dispatch` остаётся в `if:` — чтобы можно было вручную запустить деплой даже если флаг не выставлен (например, для теста после первого bootstrap).

Альтернатива (если оператор не хочет vars и готов править YAML после первого деплоя):
- Можно начать с `if: false` и убрать после первого настроенного push. Но `vars.VDS_DEPLOY_ENABLED` приятнее: один тумблер в Settings → Variables вместо правки YAML.

- Какие 4 секрета прописать в Settings → Secrets and variables → Actions → New repository secret: VDS_HOST, VDS_USER, VDS_SSH_KEY, VDS_PORT (опционально, дефолт 22)
- Какую 1 переменную прописать в Settings → Secrets and variables → Actions → Variables: VDS_DEPLOY_ENABLED=true

`appleboy/ssh-action@v1.2.5` — последний стабильный релиз на момент 2026-04-30. Не использовать `@v1` (плавающий тег — даёт неопределённость воспроизводимости).

### Шаг 5. Настроить auto-deploy через GitHub Actions

В репозитории на GitHub: **Settings → Secrets and variables → Actions**:

**Secrets** (New repository secret):
- `VDS_HOST` — IP или домен VDS
- `VDS_USER` — обычно `root` (или созданный пользователь с docker-правами)
- `VDS_SSH_KEY` — приватный SSH-ключ (full file content, начиная с `-----BEGIN OPENSSH PRIVATE KEY-----`)
- `VDS_PORT` — опционально, по умолчанию 22

**Variables** (New repository variable):
- `VDS_DEPLOY_ENABLED` = `true`

После этого каждый `git push` в `main` триггерит `.github/workflows/deploy.yml`, который через `appleboy/ssh-action` выполняет `bash /opt/tg-parser-demo/deploy.sh` на VDS.

Если `VDS_DEPLOY_ENABLED` не выставлен — workflow тихо скипается (без падения CI). Это позволяет коммитить YAML до первого деплоя и не получать красные галочки в PR.

Ручной запуск без push: GitHub → Actions → Deploy to Timeweb VDS → **Run workflow**.
